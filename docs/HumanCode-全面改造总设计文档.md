## 一、产品愿景与核心理念

### 1.1 我们在做什么

HumanCode 是一个**基于 VS Code 深度改造**的多 AI 程序员协作 IDE。它的核心价值不是替换 AI 扩展，而是在 IDE 宿主层级建立一个**"虚拟软件团队调度中心"**：

- 用户（普通人，即"老板"）只需用自然语言描述需求
- IDE 后台由多个 AI 扩展（Copilot、通义灵码等）扮演不同角色（前端、后端、测试、审查）
- IDE 充当**项目经理**，自动协调各 AI 角色的工作流转，并对用户呈现可视化的团队工作状态

### 1.2 核心技术理念

```
传统 IDE:  用户 ←→ 单个 AI 扩展  (1对1 黑盒)

HumanCode: 用户
             ↕
         [消息路由中枢]  ← 我们开发的核心
        /      |      \
   AI角色A  AI角色B  AI角色C
  (Copilot) (Lingma) (自定义)
```

**关键突破**：扩展本身保持单实例、黑盒不变，我们在 IDE 消息层面实现"虚拟多实例"，通过上下文注入让同一个扩展表现出多个独立 AI 程序员的效果。

### 1.3 已验证的技术基础（阶段一成果）

当前已完成：
- ✅ RPC 消息拦截机制（`HumanCodeRPCLogger`）
- ✅ `IExtensionMessagesService` 消息服务
- ✅ `Extension Messages View`（可查看、复制插件日志）

这证明了我们能够**透明地捕获 IDE 与任意扩展之间的全部通信**，为后续完整改造奠定了基础。

### 2.2 数据流向

```
普通用户输入需求
       ↓
AI Team Commander Panel（用户界面）
       ↓ 选择角色 + 输入任务
IAISessionManagerService
       ↓ 附加会话上下文（角色提示词 + 历史记录）
HumanCodeRPCLogger.logOutgoing() → 拦截并注入上下文
       ↓ 增强后的消息
目标 AI 扩展（如 Copilot）处理
       ↓
HumanCodeRPCLogger.logIncoming() → 拦截响应
       ↓ 记录到会话历史
IAISessionManagerService 更新状态
       ↓
UI 更新：显示响应 + AI 状态变更
       ↓ (可选) 用户触发中继
发送给下一个 AI 角色（如 Lingma QA）
```

## 三、核心技术方案

### 3.1 消息拦截与上下文注入（核心机制）

**已验证**：通过 `IRPCProtocolLogger` 接口，可以在 JSON-RPC 层透明拦截主进程与扩展之间的所有通信。

**上下文注入策略（两种，按扩展支持程度选择）**：

| 策略 | 方式 | 兼容性 | 适用场景 |
|------|------|--------|---------|
| 方案 A：参数注入 | 修改 `data.params.context` | 需扩展支持 context | Copilot 等标准扩展 |
| 方案 B：消息前缀 | 将上下文作为消息文本前缀 | 兼容所有扩展 | 降级兼容方案 |

**推荐**：优先尝试方案 A，不支持时自动降级到方案 B。

### 3.2 虚拟会话管理（Virtual Session）

单个 AI 扩展通过**不同的系统提示词（System Prompt）+ 独立历史记录**，可虚拟扮演多个不同角色。每个虚拟会话包含：

```typescript
interface ISessionContext {
    sessionId: string;           // 唯一ID
    name: string;                // 显示名 ("前端小李")
    role: string;                // 角色 ("前端开发")
    extensionId: string;         // 目标扩展 ("github.copilot")
    systemPrompt: string;        // 角色提示词
    conversationHistory: IMessage[];  // 独立对话历史
    metadata: {
        status: 'idle' | 'working' | 'waiting';
        createdAt: number;
        lastActiveAt: number;
        messageCount: number;
    };
}
```

### 3.3 AI 间消息中继（Relay）

AI 程序员间可以将输出直接作为另一个 AI 的输入：

```
AI-A 完成任务 → 用户确认 → 系统构造中继消息 → AI-B 接收并处理
```

中继消息结构：
```typescript
{
    fromSessionId: string,
    toSessionId: string,
    relayType: 'review' | 'handoff' | 'feedback',
    message: {
        originalContent: string,   // A 的输出
        instruction: string,       // 给 B 的指令
        fromRole: string           // A 的角色（提供上下文）
    }
}
```

### 3.4 全局上下文管理（阶段二）

跨会话共享项目级信息，解决"文脉遗忘"问题：

- **项目摘要**：自动提炼已完成的工作、决策记录
- **向量检索（未来）**：基于相关性从历史记录中检索上下文
- **Token 控制**：滑动窗口策略，避免上下文超出 Token 限制

### 🚧 阶段二：多 AI 会话管理（当前阶段，预计 3-4 周）

**目标**：实现多角色 AI 并行工作 + 手动消息中继。

#### 2.1 实现 `IAISessionManagerService`（第 1 周）

**技术目标**：
- 支持创建/删除/切换虚拟 AI 会话
- 独立维护每个会话的对话历史
- 向 `HumanCodeRPCLogger` 暴露当前活跃会话 ID

**验收标准**：
- 创建两个不同角色的会话，切换后发送消息，各自响应内容符合角色定义
- 历史记录独立不混淆

#### 2.2 增强 `HumanCodeRPCLogger` 支持上下文注入（第 1-2 周）

**技术目标**：
- 在 `logOutgoing` 中读取活跃会话上下文并注入到消息
- 在 `logIncoming` 中将响应自动记录到对应会话历史

**验收标准**：
- 给前端角色发送"实现登录表单"，响应中出现 HTML/CSS 相关内容
- 给后端角色发送同样问题，响应中出现 API/数据库相关内容

#### 2.3 构建 AI Team Commander Panel UI（第 2-3 周）

**技术目标**：
- 角色卡片列表（含状态指示灯）
- 任务输入区域（per-角色）
- 消息中继按钮（A → B）
- 实时对话历史展示

**验收标准**：
- 通过 UI 完成完整的"需求 → 前端实现 → QA 测试"工作流，无需命令行操作

#### 2.4 角色配置系统（第 3-4 周）

**技术目标**：
- 内置 5 种常用角色模板
- 支持用户自定义角色名称和系统提示词
- 配置持久化到 workspace settings

---

### 🚀 阶段三：自动化协作工作流（预计 4-6 周）

**目标**：用户只需描述需求，AI 团队自动协作完成开发测试循环。

#### 3.1 任务自动分发引擎

**技术目标**：
- 用户输入需求后，系统自动分析并拆分子任务
- 根据任务类型自动分配到对应角色

#### 3.2 AI 间自动中继 + 循环校验

**技术目标**：
- 开发→测试→反馈→修复 全链路自动化
- 支持设置最大循环次数和退出条件

#### 3.3 可视化工作流编辑器

**技术目标**：
- 拖拽式节点连线，定义 AI 协作流程
- 支持条件分支（如：测试通过 → 部署，否则 → 重新开发）

## 六、用户界面设计 (UX/UI)

### 6.1 设计原则

- **零门槛**：目标用户无编程背景，所有操作用自然语言描述，不暴露技术术语
- **状态透明**：AI 在做什么、做完了什么，用户随时一眼看清
- **渐进式复杂度**：基础功能极简，高级功能（如工作流编辑）按需展开
- **拟人化**：AI 角色用"成员卡片"而非"进程列表"来呈现，降低心理门槛

> 📐 详细的界面布局、元素位置、交互细节见 → [UI整体布局.md](../ui/UI整体布局.md)

---

### 6.2 AI 团队管理界面

用于创建、查看、管理所有 AI 角色成员，是整个产品的核心控制台。

**必须包含的界面元素**：

| 元素 | 作用 |
|------|------|
| **角色成员卡片** | 展示每个 AI 成员的名称、职能角色、绑定扩展、当前状态 |
| **实时状态指示** | 通过颜色/动画区分：空闲、工作中、等待确认、出错 |
| **最近对话摘要** | 在卡片上显示最近1-2条消息，无需展开也能了解进度 |
| **添加成员入口** | 随时创建新的 AI 角色，位置固定、易于发现 |
| **成员排序/分组** | 支持按角色类型或工作状态排列成员 |

---

### 6.3 角色创建与配置界面

用于定义一个新 AI 角色的身份、能力和行为方式。

**必须包含的界面元素**：

| 元素 | 作用 |
|------|------|
| **角色模板选择器** | 内置常用角色（前端/后端/测试/审查/产品经理），一键套用 |
| **角色名称输入** | 用户可自定义成员名字（如"前端小李"），增加拟人感 |
| **AI 扩展绑定选择器** | 选择该角色底层使用哪个 AI 扩展（Copilot/Lingma/...） |
| **系统提示词编辑器** | 定义角色的专业方向和行为准则，支持模板预填 + 自由编辑 |
| **角色头像/颜色选择** | 视觉区分不同角色，增强辨识度 |
| **能力标签** | 标记角色擅长的任务类型（如 React / API 设计 / 单元测试） |

---

### 6.4 任务下达界面

用于向指定 AI 角色下达具体工作任务，是用户最高频的操作入口。

**必须包含的界面元素**：

| 元素 | 作用 |
|------|------|
| **任务输入框** | 自然语言输入区域，支持多行，有占位提示（如"描述你希望完成的任务..."） |
| **目标角色指示器** | 明确显示当前在跟哪个 AI 说话，避免发错对象 |
| **历史对话区域** | 完整显示与该角色的对话记录，区分用户消息和AI消息 |
| **发送按钮** | 清晰的主要操作按钮，支持快捷键（Enter 发送） |
| **附件/上下文引用** | 可选：引用文件、代码片段、其他AI的输出作为任务背景 |
| **终止/中断按钮** | 当 AI 在处理中，提供紧急停止选项 |

---

### 6.5 消息中继界面

用于将一个 AI 的输出作为另一个 AI 的输入，实现角色间协作。

**必须包含的界面元素**：

| 元素 | 作用 |
|------|------|
| **中继触发按钮** | 在 AI 完成任务后出现，允许用户将该结果发送给另一个角色 |
| **目标角色选择器** | 指定接收方（可以是团队中任意已有成员） |
| **中继类型选择** | 明确中继意图：代码审查 / 任务移交 / 反馈修改 |
| **内容预览与编辑** | 中继前可查看、裁剪或补充要发送的内容 |

**必须包含的界面元素**：

| 元素 | 作用 |
|------|------|

| **任务状态标记** | 每条记录附带状态图标：开始 / 进行中 / 完成 / 出错 |
| **成员切换标签** | 可按成员筛选，只看某个 AI 的操作记录 |
| **导出/复制操作** | 将完整工作日志导出为文本或复制到剪贴板 |
| **清除历史入口** | 一键清空日志，开始新的工作会话 |

### 6.7 上下文管理界面

管理跨会话共享的项目背景信息，让所有 AI 成员共享同一份"项目记忆"。

**必须包含的界面元素**：

| 元素 | 作用 |
|------|------|

| **各角色会话统计** | 显示每个成员的消息数量、最近活跃时间 |

## 八、技术风险与缓解

| 风险 | 等级 | 说明 | 缓解措施 |
|------|------|------|---------|
| 上下文注入方式不被扩展接受 | 中 | 扩展可能忽略 `params.context` 字段 | 自动降级为消息前缀方式（方案B） |
| 消息识别误判 | 低 | 非 Chat 消息被误识别 | 基于 RPC 方法名精确匹配，可随时调整规则 |
| Token 超出限制 | 中 | 历史记录过长 | 滑动窗口（只保留最近N条），关键信息摘要 |
| VS Code 版本更新兼容 | 中 | RPC 层可能变动 | 只修改日志层，不改核心逻辑，降低维护成本 |
| 多会话并发冲突 | 低 | 多个 AI 同时响应导致状态混乱 | 基于 `activeSessionId` 串行处理，后续可加队列 |
| Webview 通信无法拦截 | 低 | 部分扩展用 Webview 渲染 | 阶段一聚焦 RPC 层，Webview 拦截列入阶段三研究 |
