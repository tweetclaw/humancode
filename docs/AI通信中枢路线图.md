# HumanCode: 核心技术突破与实施路线图

## 1. 当前遭遇的最大核心技术难点解析
**核心痛难点：跨扩展（独立沙盒）的消息劫持、监听与指令注入**

在当前 VS Code 的底层设计架构中，所有的扩展插件（Extension）都运行在独立的 Extension Host 进程里。像 GitHub Copilot (Codex)、通义灵码 (Lingma) 这样的独立 AI 扩展，它们是**“各自为城、互不通信”的黑盒系统**。

你所描述的最核心构想——“保留它们的独立逻辑，但 IDE 能够读写各个扩展的输入输出端口并人为控制”，从技术底层的翻译就是：**IDE 必须充当一个“超级代理人（Proxy）”和“虚拟键盘/显示器”**。我们需要在**不破坏、不修改原有第三方闭源 AI 插件的前提下**，打破它们之间的信息高墙。

技术挑战包括：
1. **如何“读”（输出拦截）**：插件生成的代码或对话通常是通过 VSCode 的内部 API（如 `vscode.chat` API、或者 WebView 渲染）直接推送给用户的。我们需要在 IDE 源码层面对这些 RPC（远程过程调用）通信或者渲染通道进行“劫持”或“镜像复制”，才能拿到 Codex 吐出的结果。
2. **如何“写”（输入注入）**：拿到结果后，由于外部插件通常不公开接收其他扩展调用的接口，我们需要通过 IDE 的底层机制“伪造人类行为”（例如：静默抓取上一步结果 -> 构造 Prompt -> 强制将其推送到 Lingma 的 Chat Input 接口中并触发执行）。

---

## 2. 实施计划与技术验证路线图 (Roadmap)

为了稳扎稳打实现这个“一人公司虚拟调度台”，我们必须按以下步骤推进核心技术的重构：

### 🏁 阶段一：基础消息通路的 I/O 劫持试验（我们在“第一步”要做的）
**目标核心**：在打通任何复杂的团队逻辑前，先证明“纯消息的中继（Relay）”是完全可行的。

*   **步骤 1 (输出捕获 POC)**：在 VS Code 核心代码库中，定位 `ChatView` 或 `Extension Host RPC` 的通信点。挂载一个监听器。当我们对着 Codex（或任何测试用 AI 插件）发消息并得到回复时，IDE 底层能成功捕获并打印这段 Response。
*   **步骤 2 (用户控制台/人工路由)**：在界面拦截这部分输出，不立即丢给代码区，而是展示在你（老板）的【工作群管理面板】里。页面提供选项：“自己修改” 或 “发给 QA 扩展测试”。
*   **步骤 3 (输入伪造 POC)**：如果你点击了“发给QA”测试，IDE 必须能够将刚才拦截的 Codex 结果组装成一段 Prompt，并通过内部 API 直接送进通义灵码 (Lingma) 等另一个插件的上下文，并强制触发该插件开始生成反馈。

**【阶段一里程碑检验标准】**：能够做到“从插件A自动接管输出，经由人的按钮确认后，作为插件B的输入投递过去”——**不打破任何扩展现有的黑盒。**

---

### 🚧 阶段二：共享存储与上下文大盘 (Shared Context & Digest)
跨插件传递消息成功后，系统立刻会面临“文脉遗忘”和“Token 容量爆炸”的挑战。插件B（测试）不知道一开始给插件A的首发需求是什么。
*   **目标**：建立跨越独立插件记忆墙的全局上下文库。
*   **行动**：在理想状态下，引入一套本地的向量库或文本摘要系统。IDE 这位“项目经理”会旁听并记录 Codex 和 Lingma 的讨论结果。系统不断提取当前的重点（“测试说缺少非空校验，代码已改回”），在下次调用任何插件时，将其作为系统提示词（System Prompt）偷偷塞入，让所有独立的 AI 插件具备“团队记忆”。

---

### 🚀 阶段三：LangChain / Agentic 自动化闭环与工作流
前期的人工接管只是体验雏形。当第一步、第二步全部做通后，就要迈入完全体的自动化节点了。
*   **目标**：IDE 后台升级为 Multi-agent 路由节点引擎框架。
*   **行动**：用户不再需要手动点击“让灵码去测试 Codex 的代码”。而是直接在图形化沙盘（Blueprint）上拖拽连线：定义【生成逻辑】节点（指定 Codex）→ 连接到【验证逻辑】节点（指定 Lingma）→ 甚至外接【循环校验】节点。IDE 内部的 Multi-Agent 网络会自动让这两个独立闭盒的 AI 工具产生互相纠错的循环对话，并只在最终“测试通过”或“卡壳报错”时，才将结果呈递给你（老板）。

---

### 总结与下一步
完全理解你的意图！你的出发点极其精准，没有好高骛远去强行重写一套大模型对话逻辑，而是**把现成最好的 AI 插件当做“外包打工人”，在 IDE 宿主级别开发一个强力的“接包管控、信息流转分发中心”**。这就是目前架构层级的真正发力点。
