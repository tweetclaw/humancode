# AI 扩展互通开发文档计划

## 开发目标与边界

- **明确范围**：当前迭代**只实现平台内的 AI 扩展互通**，不涉及 UI 自动化、焦点模拟或 OCR 等技术方案。VS Code 的扩展运行时架构设计上**不允许**扩展直接修改主界面 DOM 或执行不稳定的 UI 操作【11†L228-L232】。因此我们遵循“以平台能力实现交互”，而非依赖任何 UI hack。  
- **主机环境支持**：首阶段仅支持在**本地 Extension Host（Node.js）**上运行的端点，远程/浏览器环境作为后续扩展。VS Code 的运行时允许存在“local”（本地 Node.js）、“web”（浏览器 WebWorker）、“remote”（远端 Node.js）多种 Extension Host【1†L140-L148】【1†L162-L169】。当扩展同时可运行在本地和 Web 时，默认会优先使用 Node.js 主机【11†L215-L222】。因此，我们先在本地主机环境下实现互通逻辑，再考虑跨远端或 Web 环境的部署兼容。  
- **API 状态**：首先使用 **Proposed API** 形式定义接口，不急于发布为稳定 API。VS Code 官方文档鼓励在需求初期使用“proposed”来试验和迭代（见【11†L226-L232】稳定性说明），待方案成熟后再移入正式 API。换言之，本期迭代着重**实验可行性和设计**，不强求立即兼顾兼容性和稳定性。

## RPC 协议与 DTO 文档

AI 互通使用 VS Code 的 RPC 通信机制在主线程和扩展宿主间传输消息。我们将设计以下主要接口和数据协议：

- **Shape 接口**  
  - `MainThreadAiInteropShape`：定义主线程暴露给 ExtHost 的方法，如 `$registerEndpoint(descriptor)`, `$unregisterEndpoint(id)`, `$acceptInvocationChunk(invocationId, chunkDto)`, `$completeInvocation(invocationId, resultDto)`, `$handleInvocationError(invocationId, error)`, `$cancelInvocation(invocationId)` 等。  
  - `ExtHostAiInteropShape`：定义 ExtHost 暴露给主线程的回调，如 `$onInvocation(invocationId, requestDto)`, `$onInvocationCancel(invocationId)` 等。  

- **注册与调用 DTO**  
  - **Endpoint 注册**：`EndpointDescriptor` 包含 `id`（全局唯一）、`displayName`、`kind`（agent/tool/ui/model）、`capabilities`（如是否支持流式/工具调用）、`safety` 等字段。主线程接收后将记录端点元数据。  
  - **Invocation Start**：主线程发起调用时发送 `{ invocationId, endpointId, sessionId, request: InvocationRequest }`，包含用户输入、上下文引用等信息。  
  - **流式 Chunk**：扩展端通过 `$acceptInvocationChunk(invocationId, chunks[])` 向主线程发送一批 `InvocationChunk`。设计为批量结构以降低 RPC 调用次数，类似现有 ChatSession 批量 Progress 发送做法【turn5view4†】。  
  - **完成/错误**：扩展端调用 `$completeInvocation(invocationId, resultDto)` 或 `$handleInvocationError(invocationId, errorDto)` 标记结束，传回最终结果或错误信息。  
  - **取消**：主线程若取消调用，触发 RPC 的 Cancel 消息；在 ExtHost 创建对应 `CancellationTokenSource`。【0†63-L】（参考 VS Code 内部 `rpcProtocol.ts` 取消实现），扩展收到后应终止执行并可调用 `$handleInvocationError` 报告已取消。

- **工具调用**：在处理流式输出时，扩展可通过流中某类 `InvocationChunk`（`toolCall` 类型）发起工具调用请求，指定 `toolId` 和参数。主线程服务接收后进行仲裁（如调用 `ILanguageModelToolsService.invokeTool`【turn7view3†】），然后通过 `$acceptInvocationChunk` 发送工具调用结果块。

- **错误码与语义**：所有 RPC 通信尽量包含明确错误码和文本描述，以便前端展示和审计。鉴于主要流程均在本机内运行，我们倾向于即时失败或取消的处理，不设计网络超时机制。取消处理将遵循 `CancellationToken` 模式，如主线程通过 `rpcProtocol` 发送取消信号【11†L228-L232】。

通过明确上述协议和 DTO，主线程与扩展端的交互将如同一个“已知协议”的网络服务，避免了 ad-hoc 实现的混乱。借鉴 Chat Sessions 和 Language Model Tools 模块的实现方式（例如它们使用批量 progress dto【turn5view4†】并绑定 callId）可提高实现效率和稳定性。

## Session / Invocation 状态机文档

我们需要定义 **会话（Session）** 和 **调用（Invocation）** 的生命周期及状态，以支持错综复杂的多扩展协作场景。至少包括：

- **Session**：由主控端（或平台在内部）创建，包含 `sessionId`、`title`、`participants` 列表等。每个 Session 绑定一个 workspace/remoteAuthority，防止跨远端环境数据混合。【1†L153-L159】  
- **参与者（Participant）**：Session 的成员，可分为 `controller`（发起者）、`worker`（被调用扩展）、`ui`（聊天/视图界面）等角色。加入时需要注册端点 ID。  
- **Turn & Invocation**：会话中的每次交互可视为一个 Turn，由一个主控向某个参与者发起 Invocation（调用）。Invocation 包括一个唯一 `invocationId`，对应一次请求-响应流程。  
- **状态**：Invocation 从 **Pending**（已发送请求）进入 **Streaming**（正在接收 chunks）直到 **Completed**（成功完成）或 **Errored**（报错）或 **Cancelled**。如果扩展端崩溃或断连，Invocation 将标记为 **Orphaned**，等待恢复。  
- **取消**：当主机调用 `cancel(invocationId)` 后，对应 Invocation 进入 **Cancelled** 状态。扩展侧应响应取消并报告最终状态。  
- **恢复/重试**：如果扩展主机崩溃重启，Session Broker 允许其通过 `resumeSession(sessionId)` 拉取未完成的 Invocation。Invocation 必须设计幂等处理（例如基于 chunk 序号避免重复）才能安全重试。  

**状态图示例**（为了避免文字描述混淆，可绘制一个 Mermaid 序列图说明 `Invocation` 的请求、Chunk 返回、完成/错误流程，但由于受限这里略去示意）。清晰的状态机定义能避免恢复、取消与失败之间的交叉逻辑错误。

## 权限与安全模型文档

必须提前定义权限策略，确保扩展间通信不越权。关键原则和策略包括：

- **能力声明**：扩展在 `package.json` 中声明是否使用互通功能，比如 `aiInterop.provideEndpoint`、`aiInterop.invokeEndpoint`、`aiInterop.observeSession` 等自定义声明（参考已有 `capabilities` 体系）。只有声明了“提供端点”权限的扩展才能调用注册 API；只有声明了“调用端点”权限的扩展才能 invoke。  
- **调用授权**：默认情况下，扩展只能调用自己注册的端点。若扩展 A 企图调用扩展 B 的端点，系统应弹出授权对话框，询问用户是否允许这种跨扩展通信，并可选择“仅本次/本会话/永久授权”等（类似工具审批或网络访问授权的 UX）。【17†L136-L144】  
- **上下文共享范围**：定义 Session 内容共享级别：`none`（不共享会话上下文，仅调用指令），`lastTurn`（共享最近一次交互内容），`full`（共享整条会话历史），`redacted`（智能脱敏后共享）。仅在用户明确允许的级别下，系统才将输入历史或文件等内容传给被调用端。  
- **远端环境隔离**：如果 Session 来源于不同的 `remoteAuthority`，则**禁止直接通信**【1†L153-L159】【11†L226-L232】。例如，处在 SSH 远端的扩展与本地扩展之间的数据不得跨境传输，否则可能泄露用户隐私或超出预期信任域。  
- **危机能力审批**：对于工具调用、MCP 访问、CLI 执行等高风险功能，要复用 VS Code 的审批流程。比如，当被调用端尝试执行更改工作区的工具或启动子进程时，平台必须弹出确认框（同 `LanguageModelTools` 中对副作用工具的处理逻辑【17†L158-L166】）。用户可设置“是否提示”及持久化策略。CLI/终端操作也应考虑 `extensionKind`（只能在 Node host 上）和输入输出的安全性。  

先行定义这些策略后续很难大改，否则会影响核心设计流程，因此务必在编码前明确。权限模型直接决定了我们能否「安全」进行扩展间互通，否则后续可能返工。

## Adapter 接入规范文档

为了兼容现有生态，应为各种类型扩展定义统一接入方案（Adapter）：

- **命令适配（Command Adapter）**：将现有基于 `vscode.commands.executeCommand` 的扩展功能视作非流式工具调用。主要用于兼容旧扩展。此方式只能同步返回结果，无法进行流式输出，也无法带有复合状态。因此适用于简单任务调用，且参数类型受 VS Code 限制。adapter 实现时需验证执行参数并将结果封装为 `InvocationResult`。  
- **Chat 参与者适配（Chat Participant Adapter）**：Chat 参与者由用户通过 `@提及` 调用【14†L136-L144】。我们可以允许 Chat 参与者“显式注册”为可被平台调用的端点（例如在注册时设置 `aiInterop.callable: true`）。但默认不暴露给所有扩展，除非扩展自己声明支持互通。适配器负责监听 Chat UI 的选举调用（例如用户在 UI 中选中某个参与者），将其转化为总线的 Invocation，然后将响应片段送回 Chat UI。聊天参与者本身可能会调用工具/API，然后再返回完整回复（见【15†L1-L4】【14†L148-L151】）。  
- **工具（Tool）适配**：工具 API 本身在多扩展方案中直接复用。总线将不同扩展注册的工具视为可供 Chat 或其他 agent 调用的能力。我们无需自行实现工具调用流程，只需在总线中转发 ToolCall 并使用已有的 `ILanguageModelToolsService` 进行实际调用（参考 `MainThreadLanguageModelTools` 实现）。MCP 工具视同工具调用，走同一审批流程。Adapter 主要工作是在总线层屏蔽底层的工具类型差异。  
- **Webview 适配**：Webview 是 UI 沙箱，脚本无法访问 VS API【20†L973-L975】。我们需要设计一组消息接口，让 Webview 发送用户操作或接收系统提示。例如在 Webview 中 `acquireVsCodeApi().postMessage(...)` 向扩展发事件，扩展再通过 `panel.webview.postMessage` 推送后端反馈。适配器应负责将 Webview 的消息转换成 Session 事件，反之亦然。由于 Webview 与扩展异步通信，必须实现消息队列和超时重试等机制。用户授权需求按同一逻辑处理（某些 Webview 内容可能需要用户确认才能共享）。  
- **Terminal/CLI 适配**：通常需要 Node.js host 来执行。平台可提供内置 Adapter：在本地/远程 Node 扩展主机中启动终端或子进程，把 `stdout/stderr` 流式地作为 ChatChunk 返回给总线。适配器同样需要权限校验（不自动允许任何扩展发 shell 命令）。降级策略：如果扩展在 Web 主机，则拒绝执行；如果用户不授权，则仅返回“拒绝”消息。  

每种 Adapter 都要清晰定义**输入/输出格式**、是否支持流式/可取消，以及用户授权需求。例如，Command 只能输出一次结果、不可取消；而 CLI Adapter 则**支持流式输出且可在中途取消**（发送 SIGINT）。在文档中应逐项列出这些对比，以避免实现时的模糊和漏洞。

## 测试计划与验收标准

测试是确保功能正确的关键。建议分层测试体系并制定明确指标：

- **Unit Tests**：放在 `src/vs/workbench/services/aiInterop/**` 目录下，使用 Mocha/Jest 进行单元测试。测试内容包括会话状态机、权限策略、DTO 序列化等小模块逻辑。确保覆盖率不低于 ~80%。  
- **Integration Tests**：扩展运行级测试。创建模拟扩展（controller、worker、observer）在 Extension Development Host 中执行实测。VS Code 官方文档指出集成测试运行在特殊的“Extension Development Host”中，拥有完整 API【22†L136-L139】。测试场景包括：**单向调用**（确认基本功能）、**双向流**（大容量 chunk、检查取消是否生效）、**多扩展会话**（两个扩展交互）等。  
- **Extension Fixture Tests**：使用 VS Code Test Runner 跑真实扩展包。可以借助官方的 `@vscode/test-electron` 等工具链，验证在类似用户环境下功能正常。主要验证命令、URI、常规 API 交互是否与平台功能兼容。  
- **Multi-Host/Remote Tests**：模拟多主机环境。可以利用 Remote Development 的 SSH 容器或 Codespace 模式，创建一个“远端”工作区。同时在本地启动 VS Code 测试版并连接到该远端，在远端和本地分别运行部分扩展。测试**跨远程互通**的正确性，如远端注册的端点能否正确调用（需要 remoteAuthority 匹配），以及不匹配时是否恰当拒绝【1†L153-L159】。  

**验收指标示例**：  
- **延迟**：本地环境下消息往返（发起 Invocation 到收到最终结果）p95 < 100ms；远程环境 p95 < 300ms（网络可变情况下可适当放宽）。  
- **吞吐**：单次会话流式输出速度 >= 500 tokens/s（可通过后台模拟大量 `chunk` 测试）。  
- **取消/失败检测**：向正在进行的调用发送取消后，主线程接收确认并结束 invocation 的时间 < 200ms。  
- **安全**：远程身份不匹配的调用**必须完全被阻断**【1†L153-L159】；在没有用户授权的情况下，跨扩展调用 **不得发生**。  
- **资源占用**：长时间运行（如 5 分钟高频流场景）下主线程额外 CPU 占用 < 5%，内存增长稳定可回收。  

这些指标应写入 CI 过程，结合 VS Code 官方扩展测试指南实现自动化验证【22†L136-L139】。

## PoC 实施手册

为了快速验证核心交互，可分步推进：

1. **PoC-1：同主机注册和流式调用**。  
   - 实现 `ExtHostAiInterop.registerEndpoint` 在本地 Node 环境下注册端点，以及 `createInvocation` 逻辑；实现流式返回若干 `InvocationChunk`。  
   - 验证：使用两个简单扩展（A 调用 B 的端点），A 发起请求后 B 连续发送多条 chunk，A 能正确接收并合并。检查主线程无错误日志。  

2. **PoC-2：取消穿透**。  
   - 在 PoC-1 基础上，A 发起调用后立刻取消，检查 B 是否收到取消信号。B 在收到 CancelToken 后应终止执行并调用 `$handleInvocationError(invocationId, error)` 通知主线程。  
   - 验证：A 的 Promise 立刻 reject，主线程记录已取消；B 不再发送后续 chunk。  

3. **PoC-3：跨主机路由**。  
   - 模拟远端场景：使用本地与 remote Extension Host（可用 VS Code 的 Remote 插件），让 A 在本地发起调用，B 在远端（或反之）。  
   - 验证：调用成功时，数据往返经 main-thread 路由正常流转；如果 B 在远端但未安装（remoteAuthority 不匹配），调用应失败并报错【1†L153-L159】【11†L228-L232】。  

4. **PoC-4：权限弹窗**。  
   - 在 PoC-3 基础上，引入权限管理：A 未经授权就调用 B，应弹出用户确认对话框，拒绝则中止。实现“允许/拒绝”（使用 `IDialogService`）的 UI。  
   - 验证：用户拒绝后调用失败；选择“允许”后可执行，并可测试“只在本次”与“永久记忆”的效果。  

5. **PoC-5：工具/MCP 接入**。  
   - 实现工具调用示例：A 在流中发起 `toolCall` 请求，主线程调用 `ILanguageModelToolsService.invokeTool`。创建一个简单 tool extension（或内置工具），让其返回结果给 A。
