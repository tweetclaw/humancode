## ⚠️ 遗留问题

### 问题 1：全局上下文库的 UI 界面尚未完成

**问题描述**：
- 全局上下文库的后端功能（数据存储、检索）已实现
- 但缺少用户界面来管理全局上下文
- 用户无法通过 UI 添加、查看、编辑全局上下文

**影响**：
- 用户体验不完整
- 需要通过代码或命令行操作全局上下文

**建议解决方案**：
创建 **Task E - 全局上下文管理 UI**，包含：
1. 上下文列表面板
2. 添加/编辑/删除上下文的界面
3. 上下文分类和标签功能
4. 搜索和过滤功能

**优先级**: 中（不影响核心功能，但影响用户体验）

---

### 问题 2：Token 预算管理的自动修剪功能需要完善

**问题描述**：
- Token 预算管理的基础框架已实现
- 但自动修剪策略还不够智能
- 缺少对上下文重要性的评估机制

**当前行为**：
- 简单的 FIFO（先进先出）策略
- 可能删除重要的上下文信息

**建议改进**：
1. 实现上下文重要性评分机制
   - 使用频率
   - 最后访问时间
   - 用户标记的重要性
2. 智能修剪策略
   - 优先保留高分上下文
   - 压缩而非删除（摘要化）
3. 用户可配置的 Token 限制

**优先级**: 低（当前功能可用，但不够智能）

---

### 问题 3：大量数据时可能需要性能优化

**问题描述**：
- 测试用例 9（大量数据持久化）发现性能问题
- 100+ 条消息时，加载时间 > 3 秒
- 可能影响用户体验

**具体表现**：
- 启动时加载所有历史数据
- 没有分页或懒加载机制
- JSON 序列化/反序列化耗时

**建议优化方案**：
1. **分页加载**
   - 只加载最近 N 条消息
   - 按需加载历史消息
2. **索引优化**
   - 为消息添加索引（按时间、会话 ID）
   - 加速查询和过滤
3. **数据压缩**
   - 压缩旧消息
   - 使用更高效的序列化格式（如 MessagePack）
4. **后台加载**
   - 启动时只加载必要数据
   - 后台异步加载完整历史

**优先级**: 中（当前可用，但大量数据时体验下降）
