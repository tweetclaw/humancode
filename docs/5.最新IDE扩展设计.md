# 统一 AI IDE 扩展界面设计方案

## 1. 项目背景与目标

### 1.1 背景

当前市场上存在多个优秀的 AI IDE 扩展：
- **Claude Code** - 擅长架构设计、代码审查、复杂问题分析
- **Codex** - 擅长代码生成、自动补全、快速实现
- **Lingma** - 擅长代码测试、质量保证、bug 检测

这些扩展各有特长，但存在以下问题：
1. **界面分散**：每个扩展都有自己的 UI，用户需要在多个窗口间切换
2. **协作困难**：无法让多个 AI 协同工作，形成开发团队
3. **能力孤立**：无法根据任务特点自动调度最合适的 AI

### 1.2 项目目标

**核心目标**：构建一个统一的 AI 开发团队界面，保留各个扩展的 AI Agent 能力，实现智能任务调度和协同工作。

**具体目标**：
1. **统一界面**：一个 UI 管理多个 AI 扩展实例
2. **保留能力**：完整保留各扩展的文件读写、代码分析、工具调用等能力
3. **智能调度**：根据任务类型自动分配给最合适的 AI
4. **协同工作**：多个 AI 可以协同完成复杂任务
5. **可扩展性**：支持添加更多 AI 扩展

### 2.2 核心技术实现

#### 2.2.1 消息拦截机制

**实现位置**：`src/vs/workbench/api/browser/mainThreadWebviews.ts`

**拦截点 1：Extension Host → Webview**
```typescript
// mainThreadWebviews.ts:65-80
public async $postMessage(handle: WebviewHandle, jsonMessage: string, ...buffers: VSBuffer[]): Promise<boolean> {
    const webview = this.tryGetWebview(handle);
    if (!webview) {
        return false;
    }
    const { message, arrayBuffers } = deserializeWebviewMessage(jsonMessage, buffers);

    // 拦截发送到 AI 扩展的消息
    const viewType = (webview as any).providedViewType;
    if (this.isAIExtension(viewType)) {
        // 转发到统一 UI
        await this.forwardToUnifiedUI(handle, message, 'toWebview');
    }

    return webview.postMessage(message, arrayBuffers);
}
```

**拦截点 2：Webview → Extension Host**
```typescript
// mainThreadWebviews.ts:82-97
private hookupWebviewEventDelegate(handle: WebviewHandle, webview: IWebview, options) {
    disposables.add(webview.onMessage((message) => {
        // 拦截来自 AI 扩展的消息
        const viewType = (webview as any).providedViewType;
        if (this.isAIExtension(viewType)) {
            // 转发到统一 UI
            this.forwardToUnifiedUI(handle, message.message, 'fromWebview');
        }

        const serialized = serializeWebviewMessage(message.message, options);
        this._proxy.$onMessage(handle, serialized.message, new SerializableObjectWithBuffers(serialized.buffers));
    }));
}
```

#### 2.2.2 AI 扩展识别

```typescript
// 识别 AI 扩展的 viewType
private isAIExtension(viewType: string | undefined): boolean {
    if (!viewType) return false;
    
    const aiExtensions = [
        'claude',           // Claude Code
        'codex',            // GitHub Codex
        'lingma',           // Alibaba Lingma
        'copilot',          // GitHub Copilot
        'codegeex',         // CodeGeeX
        'tabnine',          // Tabnine
    ];
    
    return aiExtensions.some(ext => viewType.toLowerCase().includes(ext));
}
```

#### 2.2.3 消息转发到统一 UI

```typescript
// 消息转发机制
private async forwardToUnifiedUI(
    handle: WebviewHandle, 
    message: any, 
    direction: 'toWebview' | 'fromWebview'
): Promise<void> {
    // 获取统一 UI 的 webview 实例
    const unifiedUI = this.getUnifiedUIWebview();
    if (!unifiedUI) return;

    // 构造转发消息
    const forwardedMessage = {
        type: 'ai-extension-message',
        direction,
        sourceHandle: handle,
        sourceViewType: this.getViewType(handle),
        timestamp: Date.now(),
        payload: message
    };

    // 发送到统一 UI
    await unifiedUI.postMessage(forwardedMessage);
}
```

### 2.4 关键技术挑战与解决方案

#### 2.4.1 挑战 1：消息格式不统一

**问题**：不同 AI 扩展的消息格式可能不同

**解决方案**：
```typescript
// 消息适配器
class MessageAdapter {
    // 标准化消息格式
    normalize(message: any, sourceType: string): StandardMessage {
        switch (sourceType) {
            case 'claude':
                return this.normalizeClaudeMessage(message);
            case 'codex':
                return this.normalizeCodexMessage(message);
            case 'lingma':
                return this.normalizeLingmaMessage(message);
            default:
                return message;
        }
    }
    
    // 反标准化消息
    denormalize(message: StandardMessage, targetType: string): any {
        switch (targetType) {
            case 'claude':
                return this.denormalizeToClaudeMessage(message);
            case 'codex':
                return this.denormalizeToCodexMessage(message);
            case 'lingma':
                return this.denormalizeToLingmaMessage(message);
            default:
                return message;
        }
    }
}
```

#### 2.4.2 挑战 2：Extension Host 能力保留

**问题**：如何确保 AI 扩展的文件读写、工具调用等能力不受影响

**解决方案**：
- **不修改 Extension Host 层**：所有 AI 扩展的核心逻辑保持不变
- **透明转发**：消息拦截后仍然转发给原始的 Extension Host
- **双向通信**：统一 UI 可以接收消息副本，但不阻断原始通信

```typescript
// 透明转发机制
private async forwardToUnifiedUI(handle: WebviewHandle, message: any, direction: string): Promise<void> {
    // 1. 转发到统一 UI（不阻塞）
    this.sendToUnifiedUI(handle, message, direction).catch(err => {
        console.error('Failed to forward to unified UI:', err);
    });
    
    // 2. 继续原始消息流（不受影响）
    // 原始的 postMessage 或 onMessage 处理继续执行
}
```

#### 2.4.3 挑战 3：多实例管理

**问题**：如何管理多个 AI 扩展实例

**解决方案**：
```typescript
// AI 实例管理器
class AIInstanceManager {
    private instances = new Map<string, AIInstance>();
    
    // 注册 AI 实例
    registerInstance(handle: string, viewType: string, webview: IWebview): void {
        const instance: AIInstance = {
            handle,
            viewType,
            webview,
            status: 'idle',
            currentTask: null
        };
        
        this.instances.set(handle, instance);
    }
    
    // 获取可用实例
    getAvailableInstance(aiType: string): AIInstance | null {
        for (const [handle, instance] of this.instances) {
            if (instance.viewType.includes(aiType) && instance.status === 'idle') {
                return instance;
            }
        }
        return null;
    }
    
    // 发送消息到特定实例
    async sendToInstance(handle: string, message: any): Promise<boolean> {
        const instance = this.instances.get(handle);
        if (!instance) return false;
        
        return instance.webview.postMessage(message);
    }
}
```

## 4. 技术风险与应对

### 4.1 风险 1：消息格式兼容性

**风险描述**：不同 AI 扩展的消息格式可能差异很大，难以统一

**应对措施**：
- 设计灵活的消息适配器架构
- 为每个 AI 扩展编写专门的适配器
- 使用标准化的中间消息格式

### 4.2 风险 2：性能开销

**风险描述**：消息拦截和转发可能带来性能开销

**应对措施**：
- 使用异步非阻塞的消息转发
- 实现消息批处理
- 添加性能监控和优化

### 4.3 风险 3：扩展兼容性

**风险描述**：AI 扩展更新可能破坏兼容性

**应对措施**：
- 设计版本检测机制
- 实现向后兼容的适配器
- 定期测试主流 AI 扩展的新版本
