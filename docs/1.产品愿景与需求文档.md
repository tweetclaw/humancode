# HumanCode: 产品愿景与核心需求文档 (PRD)

## 1. 原理与定位 (First Principles & Positioning)
- **为谁设计**：为缺乏编程背景，但具备创新驱动力的非专业人士（创始人、设计师、业务专家、“超级个体”）量身打造。
- **核心形态**：抛弃“人手键盘敲击代码”的落后范式。这是一个**集成了多角色 AI 的“可视化虚拟软件团队”**。
- **终极愿景**：**我们的 IDE 就是一人公司的专属开发团队**。你不是在这个软件里“写代码”，而是在“当老板、带团队”。

---

## 2. 核心价值主张 (Core Value Proposition)
现有的 IDE（包括 VSCode 及其衍生品）无论拥有多强大的 AI 辅助，其本质依然是“服务于专业程序员的代码修改器”。它们要求使用者**从机器和编译器的逻辑去理解世界**（认识文件树、理解环境依赖、读懂终端报错日志）。

HumanCode 彻底颠覆了这一点。对于非专业人士，**语法是包袱，环境是门槛，业务逻辑才是价值**。IDE 内置了一支具备极高专业素养、7x24小时随时待命的虚拟研发团队。使用者只需要具备“产品的宏观把控能力”，通过“可视化拼图 + 团队指令（自然语言）”即可完成全套开发。一人即一公司，一台电脑即一整个研发中心。

---

## 3. 核心需求与形态设计 (Core Feature Requirements)

### 3.1 界面重构：从“代码编辑器”转向“项目作战室 (War Room)”
传统的 IDE 默认映入眼帘的是闪烁的光标和代码文本框。但在 HumanCode 中，主视觉是一块**可视化沙盘**：
- **UI 预览优先**：用户看到的是应用的所见即所得 (WYSIWYG) 页面、组件结构和用户流转图。
- **对话即开发**：用户选中某个空白区域或现有组件，可以直接输入语音/文字：“让这个登录表单支持微信扫码，并调整得更有现代感”。
- **过程被折叠**：没有难以看懂的红黄报错底纹。用户发号施令，虚拟员工在后台沟通与编码，最终将完成的 UI 直接反馈在画板上。

### 3.2 自行运移的 AI 代理团队 (Virtual Agentic Team)
这不是单一的”AI 对话框”，而是基于多模态大模型的**职能分化型 Agent 矩阵**：
- **虚拟产品经理 (PM Agent)**：应对模糊需求。用户说”我要做个小红书一样的应用”，PM 负责拆解需求、建立任务看板，并转化为结构化的产品定义。
- **虚拟设计师 (UI/UX Agent)**：随时根据用户反馈生成高保真界面，免去手写 CSS 和繁琐排版。
- **虚拟工程师 (Frontend & Backend Agents)**：在底层默默搭建框架、生成代码、维护技术栈；遇到技术难点，前后端会自动进行 Multi-Agent 协作讨论修复。
- **虚拟运维 (DevOps Agent)**：接管最为令人头疼的环境配置与发布逻辑。

#### 3.2.1 IDE 的边界与职责

**IDE 提供的是”消息聚合与转发平台”，不是”智能决策系统”**：

**IDE 负责的核心功能**：
- ✅ **消息汇总**：将所有 AI 角色的消息集中展示在统一面板
- ✅ **消息路由**：将用户的指令准确传递给选中的 AI 角色
- ✅ **消息转发**：支持将一个 AI 的输出转发给另一个 AI
- ✅ **会话隔离**：每个 AI 角色只看到发给它的消息，保持上下文清晰
- ✅ **上下文共享**：提供全局上下文库，让所有 AI 共享项目记忆

**IDE 不负责的功能**：
- ❌ **需求分析**：这是 AI 项目经理的工作，不是 IDE 的工作
- ❌ **任务拆分**：AI 项目经理根据需求自主拆分任务
- ❌ **角色分配**：AI 项目经理决定将任务分配给哪个 AI 角色
- ❌ **智能决策**：所有决策由 AI 自己完成，IDE 不做关键词匹配或规则引擎

**为什么这样设计**：
1. **AI 的能力远超关键词匹配**：让 AI 项目经理自己分析需求、拆分任务，比 IDE 的规则引擎更智能
2. **保持系统简单性**：IDE 只需提供基础设施，不越界做 AI 的工作
3. **用户控制任务调度**：第一版让用户通过消息转发控制任务流转，确保可靠性

**工作流程示例**：
```
用户 → 项目经理：”实现用户登录功能”
    ↓
项目经理分析需求，回复：”我会安排前端工程师实现登录页面...”
    ↓
用户点击”转发”按钮，将项目经理的回复转发给前端工程师
    ↓
前端工程师收到任务，开始实现
    ↓
前端工程师完成后，用户将结果转发给测试工程师
```

在这个流程中，IDE 只负责消息的展示和转发，所有的任务分析、拆分、协调都由 AI 项目经理完成。

### 3.3 代码防滚与黑盒化管理
- **不可见的复杂性**：底层文件树、包管理（npm/pip）、Git 历史对普通用户是折叠的黑盒。
- **极客模式 (Geek Mode) 的保留**：作为一款优秀的 IDE 底座，如果有跨界的高手或者雇佣了真实的极客介入项目，可以一键开启极客模式。黑盒展开，恢复完整的 VSCode 文本编辑专业级能力，保证产品上限不被锁死。

### 3.4 “零阻力”云原生运行环境
所有的本地环境灾难（如 Node 版本不兼容、环境变量未配置）在 HumanCode 都是不应该存在的：
- 按下“运行”键，直接在沙盒或浏览器容器环境（如 WebContainers 技术）中呈现应用。没有冗长的终端流水线，只有友好的进度条反馈。
- **一键全球发布**：对接主流 Serverless 或边缘端部署，从创意到上线的路径只有一步。

## 4. 技术演进路线 (针对现存 VS Code 底座的改造计划)
基于现有的 VS Code 二次开发，我们需要采取“削弱与重构并重”的策略：

1. **GUI 层面大换血**：隐藏绝大多数右下角的系统状态栏、隐蔽终端面板（Terminal 面板设为默认关闭或深度隐藏），修改左侧 `Explorer` 文件树为产品结构树（`Pages`, `Components`, `Database`）。
2. **中心画板注入**：将核心的 Monaco 文本编辑器替换成自定义的可视化 Node 流 / 画布插件体系为主视图。文本编辑作为后备功能降级。
3. **AI 通信中枢建设**：新增核心业务面板——**团队工作群 (Team Chat)**。底层需要对接类似 LangChain / OpenAI 的 Multi-agent 框架，实现 Agent 之间的上下文传递和决策共享。
4. **事件流拦截**：拦截普通的 Git 行为、保存操作、构建操作，将其封装为极其简单的白话术语按钮（例如“保存当前版本”、“提交给测试”、“对外发布”）。

---

## 5. 结语 (Conclusion)

每一次功能的增删、每一张界面的排列，都必须问自己一个问题：**“如果我是一位老板，我在指挥团队时需要关心这个底层细节吗？”** 如果答案是否，就不该让它出现在用户的认知视野中。

**HumanCode 不在一堆现有 IDE 中卷性能和语法插件，而是要推动软件工程向非技术人群的全面平权。**
