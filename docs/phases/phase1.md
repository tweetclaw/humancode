# Phase 1 — 多 AI 虚拟会话管理

**状态**: 🏗️ 进行中  
**预计周期**: 3-4 周  
**前置依赖**: Phase 0 ✅

---

## 一、阶段目标

让单个 AI 扩展（如 Copilot）能够通过**虚拟会话 + 角色提示词**的方式，同时扮演多个独立的 AI 程序员角色。用户可以通过一个统一的 UI 面板管理这些角色、分配任务，并手动触发角色间的消息中继。

**本阶段结束后，用户应当能够**：
- 在 IDE 中创建多个 AI 角色（如"前端工程师"、"QA 测试"）
- 向特定角色发送任务，角色根据其设定进行响应
- 看到每个角色的工作状态（空闲/工作中/等待）
- 手动将一个 AI 的输出中继给另一个 AI

## 三、整体目标（技术方向）

> ⚠️ 本节为阶段目标概述，具体实施方案将在阶段开始时展开讨论。

### 3.1 虚拟会话管理
- 每个 AI 角色对应一个独立的"虚拟会话（Session）"
- 会话包含：角色名称、角色提示词（System Prompt）、独立对话历史
- 多个会话可以同时存在，但同一时刻只有一个"活跃会话"接收消息

### 3.2 上下文注入
- 当用户向某个角色发送消息时，系统自动将该会话的角色提示词 + 历史记录附加到消息中
- AI 扩展收到的消息包含完整上下文，因此响应会符合角色设定
- 两种注入策略（参数注入 / 消息前缀），根据扩展支持程度自动选择

### 3.3 AI Team Commander Panel UI
- 左侧边栏独立入口，展示所有 AI 角色卡片
- 每张卡片显示：角色名、状态指示灯、最近对话摘要、操作按钮
- 支持创建新角色（向导式）、发送任务、中继消息

### 3.4 角色配置
- 内置 5 种常用角色模板（前端、后端、测试、代码审查、产品经理）
- 支持用户修改角色名称和自定义提示词
- 配置持久化，重启 IDE 后保留

## 四、验收标准

> ⚠️ 详细测试用例将在实施过程中补充。

| 验收项 | 预期结果 |
|-------|---------|
| 会话隔离测试 | 向前端角色和后端角色发同一个问题，回答内容明显不同 |
| 上下文连续性测试 | 在同一会话内的后续提问，AI 能记住前面的对话内容 |
| 消息中继测试 | 点击"中继"后，AI-B 收到的消息中包含 AI-A 的输出 |
| UI 状态同步测试 | 任务进行中显示工作状态，完成后恢复空闲状态 |
| 配置持久化测试 | 重启 IDE 后，之前创建的角色仍然存在 |
