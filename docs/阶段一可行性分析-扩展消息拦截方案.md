## 执行摘要

**结论：完全可行** ✅

## 一、核心发现

### 1.1 VS Code 的扩展通信架构

VS Code 使用了清晰的 **RPC (Remote Procedure Call)** 架构来实现主进程与扩展进程之间的通信:

```
┌─────────────────┐         RPC Protocol          ┌──────────────────┐
│  Main Process   │ ◄──────────────────────────► │ Extension Host   │
│  (Workbench)    │    (IMessagePassingProtocol)  │   (Extensions)   │
└─────────────────┘                                └──────────────────┘
         │                                                   │
         │                                                   │
    ┌────▼────┐                                        ┌────▼────┐
    │ Main    │                                        │  Ext    │
    │ Thread  │                                        │  Host   │
    │ Proxies │                                        │ Proxies │
    └─────────┘                                        └─────────┘
```

**关键文件位置:**
- RPC 协议核心: [src/vs/workbench/services/extensions/common/rpcProtocol.ts](src/vs/workbench/services/extensions/common/rpcProtocol.ts)
- Extension Host 管理器: [src/vs/workbench/services/extensions/common/extensionHostManager.ts](src/vs/workbench/services/extensions/common/extensionHostManager.ts)

### 1.3 IRPCProtocolLogger 接口

在 [rpcProtocol.ts:107-110](src/vs/workbench/services/extensions/common/rpcProtocol.ts#L107-L110) 中定义:

```typescript
export interface IRPCProtocolLogger {
    logIncoming(msgLength: number, req: number, initiator: RequestInitiator, str: string, data?: any): void;
    logOutgoing(msgLength: number, req: number, initiator: RequestInitiator, str: string, data?: any): void;
}
```

这个接口会捕获:
- **消息方向**: Incoming (扩展→主进程) / Outgoing (主进程→扩展)
- **消息长度**: 字节数
- **请求 ID**: 用于追踪请求-响应对
- **发起方**: LocalSide / OtherSide
- **消息类型**: 如 `receiveRequest`, `reply`, `ack` 等
- **消息数据**: 完整的参数和返回值

### 1.4 现有的 RPCLogger 实现

在 [extensionHostManager.ts:505-538](src/vs/workbench/services/extensions/common/extensionHostManager.ts#L505-L538) 中:

```typescript
class RPCLogger implements IRPCProtocolLogger {
    private _totalIncoming = 0;
    private _totalOutgoing = 0;

    logIncoming(msgLength: number, req: number, initiator: RequestInitiator, str: string, data?: any): void {
        this._totalIncoming += msgLength;
        this._log('Ext → Win', this._totalIncoming, msgLength, req, initiator, str, data);
    }

    logOutgoing(msgLength: number, req: number, initiator: RequestInitiator, str: string, data?: any): void {
        this._totalOutgoing += msgLength;
        this._log('Win → Ext', this._totalOutgoing, msgLength, req, initiator, str, data);
    }

    private _log(direction: string, totalLength: number, msgLength: number, req: number,
                  initiator: RequestInitiator, str: string, data: any): void {
        // 格式化并输出到 console.log
        console.log(`[${direction}][${totalLength}][len: ${msgLength}]${req} - ${str}`, data);
    }
}
```

## 五、技术风险评估

### 5.1 风险等级: 低 ✅

| 风险项 | 等级 | 说明 | 缓解措施 |
|--------|------|------|----------|
| 性能影响 | 低 | 日志记录可能增加少量开销 | 使用异步日志,避免阻塞主线程 |
| 兼容性 | 低 | 基于稳定的 RPC 协议 | VS Code 的 RPC 协议已使用多年,非常稳定 |
| 维护成本 | 中 | VS Code 更新可能需要适配 | 我们只修改日志层,不改核心逻辑 |
| 扩展兼容性 | 无 | 不影响任何现有扩展 | 完全透明,扩展无感知 |

### 5.2 已知限制

1. **Webview 通信**
   - 扩展的 Webview 使用独立的消息通道
   - 需要额外的拦截点 (但不影响阶段一)

2. **Language Server Protocol**
   - LSP 使用独立的 JSON-RPC 通道
   - 需要在 LSP 客户端层面拦截

3. **扩展间直接通信**
   - 扩展可以通过 `vscode.extensions.getExtension()` 直接调用
   - 这种通信不经过 RPC 协议,无法拦截
